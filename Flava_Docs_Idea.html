1. 배경02 / 09
Background

서비스 확장에 맞춰, 문서도 조금 더 찾기 쉬운 구조가 필요

01

상황

  • Flava Cloud의 서비스 확장과 함께 문서의 양과 구조가 빠르게 증가할 것으로 예상
  • 대규모 프로젝트를 포함한 다양한 유형의 사용자가 유입되는 시점
  • 많은 문서와 기능을 일관된 구조로 재정립해 손쉽게 서비스 사용까지 도달할 필요
문서가 많아질수록, 어디서 시작하면 되는지를 더 쉽게 보여줄 필요가 있음

사용자 니즈

처음 사용하는 사용자

시작 지점을 빠르게 안내받고, 최소한의 단계로 서비스와 연결되어야 함

경험이 있는 사용자

스펙, 제약 사항, API를 정확하게 탐색하고, 확장된 사용 아이디어까지 얻을 수 있어야 함

대규모/크리티컬 프로젝트 사용자

운영, 확장, 보안, 제약 사항을 정확하고 일관된 프로세스로 확인할 수 있어야 함

2. 방향03 / 09
Direction

Flava Docs는 Self-Service 기반 지식 플랫폼을 지향

“사용자 환경에 맞게 진화하는 Private AI Cloud를 위한 성장형 지식 플랫폼”

2.1 Self-Service Docs

개인 개발자부터 미션 크리티컬한 대규모 프로젝트까지, 사용자 수준에 맞는 진입 경로와 활용 가이드를 제공

2.2 함께 진화하는 Docs

Flava AI Assistant, 사용자 커뮤니티, Use Case 확장을 통해 질문·답변·문서가 자연스럽게 이어지는 구조로 발전

2.3 Private Cloud 반영

보안 정책, 제약 사항, 내부 프로세스를 포함하여 실제 운영 환경까지 고려한 지식 제공

AI Assistant Community Use Case Security & Governance Operator Guide
질문 유형 인사이트04 / 10
Why This Matters

클라우드에서 가장 많은 질문은
기능 설명보다 상황 판단에 가깝다

자주 나오는 질문

  • 언제 / 왜 써야 하나요?
    예: Server vs Function, Kubernetes가 꼭 필요한가요?
  • 어떻게 구성해야 하나요?
    예: 네트워크 구조, 운영 구조, 서비스 조합
  • 왜 안 되나요?
    예: 권한 에러, 연결 실패, 배포 후 동작 이슈

자주 부족해지는 정보

  • 이게 맞는 구성인가요?
    보안, 비용, 운영 관점의 검증 필요
  • 기존 시스템에서 어떻게 옮기나요?
    온프레미스 / 타 클라우드에서의 마이그레이션
  • 실제 서비스는 어떻게 만드나요?
    Use Case, 아키텍처 예시, 운영 팁
Insight

클라우드에서 가장 많은 질문은 “이 기능이 무엇인가”보다
“내 상황에서 어떻게 써야 하는가”에 가깝습니다.

3. 현재 구조04 / 09
Current State

상품 구조는 잘 정리되어 있음.
이제 필요한 것은 사용자 관점의 안내

02

현재 강점

  • Management / IAM / Compute & Container / Network / Storage / Database / Security / AI 등 카테고리 구조가 명확함
  • 상품 단위 설명에는 강점이 있음
  • 서비스 구조 기반 확장 가능성이 충분함
첫 화면의 전체 상품 카테고리 구조

보완 포인트

  • 초보 사용자는 어디서 시작해야 하는지 판단이 어려움
  • 브레드크럼과 사이드바가 상품/문서 중심이라 서비스 전체 흐름 파악이 어려움
  • 기존 상품별 Getting Started가 존재해도, 사용자 상황별 시작점 연결은 약함
기존 Server 문서: 상품 단위 탐색에는 강점, 전체 서비스 관점 탐색은 제한적

3.2 AI Assistant07 / 09
3.2 AI-Connected Docs

Docs와 AI Assistant를 연결해 질문 기반 탐색으로 전환

Docs ↔ AI 연결

  • Docs 내 AI 진입 포인트 위치 정의
  • 페이지 문맥 기반 질문 흐름 설계
  • 상품 Overview에서 콘솔/가이드/AI로 자연스럽게 연결

FAQ 활용

  • 상품 내 개별 존재하는 FAQ와 가이드 구조 점검
  • AI Search / FAQ / 운영 / 마이그레이션 가이드까지 연결 가능한 기반 정리
  • 반복 질문을 우선 연결하고, 필요한 내용은 Docs로 다시 정리

Troubleshooting

  • 주요 질문/에러 케이스를 정의하고 대응 가이드를 정리
  • AI를 활용해 문제 원인 파악부터 해결 가이드 탐색까지 연결
  • 장기적으로는 문제 해결 이후 필요한 액션까지 이어질 수 있는 구조 검토

정적 문서 검색을 넘어서, 질문 → 답변 → 관련 Docs 연결의 흐름을 만든다.

Community & Operator

커뮤니티와 운영자 콘텐츠를 통해 지식이 축적되고 순환되는 구조 형성

3.3 Flava User 커뮤니티 체계 마련

  • AWS re:Post와 유사하게 복잡한 사용 방법에 대한 사용자 간 질문과 Use Case 공유 활성화
  • 커뮤니티 채널 정의 및 운영 방식 협의 (리스크 검토 포함)
  • 반복 질문과 VOC를 Docs / Use Case / AI 응답 품질 개선으로 연결

핵심 액션

주요 질문 유형 정리 · VOC 수집 기준 정의 · 반복 질문 Docs 반영 · Use Case 콘텐츠 추가

3.4 운영자를 위한 콘텐츠 추가

  • 운영 시 고려사항 중심의 콘텐츠 보완
  • 확장 시나리오와 관련 문서 연결
  • 권한, 제약 사항, 보안 기준을 One-stop으로 연결
  • 기존 환경에서 Flava로 자연스럽게 넘어올 수 있도록 심리스한 마이그레이션 지원 강화

핵심 액션

운영 시나리오 정의 · API/운영 툴 체크포인트 정리 · 정책 항목 정의 · WFPF 연계 프로세스 검토

4. 기대 효과09 / 09
Expected Value

문서를 더 많이 만드는 것이 아니라,
필요한 지식이 더 빨리 닿고 더 잘 축적되는 구조를 만드는 것

사용자

  • 어디서 시작해야 하는지 더 빠르게 이해
  • AI와 Docs를 오가며 스스로 해결하는 경험 강화
  • 단일 상품 사용에서 전체 서비스 이해로 자연스럽게 확장

운영자

  • 반복 문의를 FAQ·가이드·AI 응답으로 흡수
  • 운영/마이그레이션 지식을 문서와 함께 축적
  • 내부 지원 리소스를 더 필요한 영역에 집중

비즈니스

  • 신규 사용자 온보딩 부담 완화
  • 서비스 활용도와 확장 사용 가능성 확대
  • 커뮤니티·AI·Docs가 함께 축적되는 장기 자산 확보
Wrap-up

Flava Docs는 사용설명서를 정리하는 작업을 넘어,
사용자 질문, AI 응답, 커뮤니티 경험, 운영 지식이 자연스럽게 연결되는 기반이 될 수 있습니다.
